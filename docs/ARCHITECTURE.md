# labrat: Architecture

## The Three Layers

```
┌─────────────────────────────────────────────┐
│  Layer 1: INFRASTRUCTURE (immutable)        │
│  Your eval pipeline: data → model → metrics │
│  The AI cannot modify this.                 │
├─────────────────────────────────────────────┤
│  Layer 2: EXPERIMENTS (AI-generated)        │
│  Config YAMLs: what to test this cycle      │
│  Single-delta from parent. AI generates.    │
├─────────────────────────────────────────────┤
│  Layer 3: CONTROL (the orchestrator)        │
│  orchestrator.md: how to behave             │
│  constitution.md: how to score              │
│  branches.yaml: what to explore             │
│  State files: what happened so far          │
└─────────────────────────────────────────────┘
```

### Layer 1: Infrastructure
Your existing code. The AI calls it but can't modify it. This prevents
the AI from gaming the evaluation metric.

### Layer 2: Experiments
Config files generated by the AI each cycle. Each config changes ONE parameter
from its parent (single-delta rule). The AI's creativity is channeled through
constrained search.

### Layer 3: Control
Markdown and YAML files that define the AI's behavior. The orchestrator is
the "agent brain" -- it tells the AI what to do step by step. The constitution
defines scoring rules. Branches define the search space.

## State Management

```
state/
├── cycle_counter.json     # Current cycle, total experiments
├── branch_beliefs.json    # Per-branch: EV, uncertainty, last explored
├── champions.json         # Best experiment per branch
├── budget.json            # Remaining compute credits per branch
└── experiment_log.jsonl   # Append-only history of all experiments
```

**No database. No external service.** The AI reads JSON, makes decisions,
writes JSON. State is human-readable, diffable, and git-trackable.

## The Scoring Pipeline

```
Experiment Result
    │
    ▼
Hard Gates (binary pass/fail)
    │ metric > baseline?
    │ p-value < 0.10?
    │ model not collapsed?
    │
    ▼ (if all pass)
Soft Score (0-1 composite)
    │ D: deployment metric (0.40)
    │ R: robustness (0.25)
    │ G: generalization (0.15)
    │ C: calibration (0.10)
    │ I: information gain (0.10)
    │ K: complexity penalty (0.05)
    │
    ▼
Verdict
    │ PROMOTE: score >= champion
    │ MARGINAL: within 0.05 of champion
    │ REJECT: below champion - 0.05
```

## The Allocator

UCB1-inspired branch selection balances exploration (try new branches)
vs exploitation (revisit productive branches).

```
priority = 0.3 * EV + 0.4 * uncertainty + 0.3 * recency

EV = exponential moving average of improvement rate
uncertainty = 1 / sqrt(1 + n_experiments)  [decays with exploration]
recency = min(1, cycles_since_last / 3)    [rewards waiting]
```

The allocator naturally transitions from exploration (early cycles,
high uncertainty) to exploitation (later cycles, productive branches).

## The Loop

```
Human: "Read orchestrator.md and execute one cycle"
  │
  ▼
AI reads state files (30 seconds)
  │
  ▼
AI picks branch via allocator formula (instant)
  │
  ▼
AI generates experiment config (1 minute)
  │
  ▼
Proxy run (5-30 minutes)
  │ Kill if obviously bad
  ▼
Confirm run (30-120 minutes)
  │
  ▼
Judge scores mechanically (instant)
  │
  ▼
AI updates state files (1 minute)
  │
  ▼
AI writes handoff for next cycle
  │
  ▼
/loop fires again in 1 hour
```

## Hooking Into Different Systems

### Claude Code
Native `/loop` command. The primary target.

### Cursor / Windsurf / Other AI IDEs
Most have some form of "run this prompt repeatedly."
The orchestrator.md works with any system that can read files + run shell commands.

### Headless (cron + API)
```python
# run_one_cycle.py
import anthropic

client = anthropic.Anthropic()
orchestrator = open("research_lab/orchestrator.md").read()

response = client.messages.create(
    model="claude-sonnet-4-20250514",
    max_tokens=8000,
    system=orchestrator,
    messages=[{"role": "user", "content": "Execute one research cycle."}],
    tools=[...],  # file read/write/bash tools
)
```

Run via cron: `0 * * * * python run_one_cycle.py >> lab.log 2>&1`

### GitHub Actions
```yaml
name: Research Cycle
on:
  schedule:
    - cron: '0 * * * *'  # hourly
jobs:
  cycle:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: python run_one_cycle.py
      - run: git add research_lab/state/ research_lab/logs/
      - run: git commit -m "Lab cycle $(cat research_lab/state/cycle_counter.json | jq .cycle)"
      - run: git push
```

## Parallel Branch Execution

Branches are independent. The orchestrator exploits this by running experiments across branches concurrently.

### Why Parallel by Default

A cycle that explores 4 branches sequentially takes 4x as long as one that runs them in parallel. Since each branch experiment is fully independent (different config, different output dir, different judge invocation), there is zero conflict during execution. The only serial bottleneck is the state update at the end.

### The Parallel Model

```
Orchestrator reads state
    │
    ├── Select N branches by priority
    ├── Generate N configs (one per branch)
    ├── Write active_agents.json (dashboard shows pulsing dots)
    │
    ├── Launch N subagents IN PARALLEL ─────────────────────────┐
    │     ├── Agent: features branch ──► run_experiment + judge │
    │     ├── Agent: model branch ──► run_experiment + judge     │ concurrent
    │     ├── Agent: objectives branch ──► run_experiment + judge│
    │     └── Agent: preprocessing branch ──► run_experiment + judge
    │                                                            │
    ├── Collect all results ◄───────────────────────────────────┘
    │
    ├── Update state SEQUENTIALLY (shared files)
    ├── Clear active_agents.json
    └── Write handoff
```

### In Claude Code

Each branch maps to one `Agent` tool call. Launch them in a single message:

```
Agent(name="features-branch", prompt="Run experiment features_bigram_c2...", mode="bypassPermissions")
Agent(name="model-branch", prompt="Run experiment model_catboost_c1...", mode="bypassPermissions")
Agent(name="objectives-branch", prompt="Run experiment objectives_balanced_c2...", mode="bypassPermissions")
```

Claude Code runs these concurrently. Each subagent:
1. Creates experiment directory and config
2. Runs `run_experiment.py`
3. Runs `judge.py`
4. Returns RESULT + VERDICT lines

The orchestrator collects results, then updates state files one at a time.

### What Parallelizes and What Doesn't

| Step | Parallel? | Why |
|------|-----------|-----|
| Read state | No | Single read at cycle start |
| Select branches | No | Depends on state |
| Generate configs | Yes | Independent per branch |
| Run experiments | **Yes** | Fully independent |
| Run judge | **Yes** | Independent per experiment |
| Update state | **No** | Shared files (champions, budget, beliefs) |
| Write handoff | No | Single summary |

### Sequential Fallback

If the runtime doesn't support subagents (Cursor, headless cron, etc.), the orchestrator falls back to sequential execution. The cycle still works, just N times slower.

## Live Dashboard

The dashboard (`research_lab/dashboard.html`) is a zero-dependency HTML file served by `python -m http.server`. It auto-refreshes every 30 seconds and reads state files via XHR.

### Agent Status Indicators

The dashboard reads `state/active_agents.json` to show which branches have live subagents:

```json
{
  "agents": {
    "model": {"status": "running", "experiment_id": "model_lr001_c5", "started_at": "..."},
    "features": {"status": "running", "experiment_id": "features_trigram_c3", "started_at": "..."}
  }
}
```

Each branch row shows:
- Pulsing black dot + experiment ID when a subagent is running
- Gray dot + "idle" when no agent is active

The orchestrator writes this file before launching subagents and clears it after collecting results.

### Setup

The orchestrator starts the dashboard automatically on first run. Manual start:
```bash
cd research_lab && python -m http.server 8787 &
# Open http://localhost:8787/dashboard.html
```

## User Onboarding

On the first cycle (cycle_counter.json shows cycle=0), the orchestrator asks the user:

1. **Parallelism**: "How many branches per cycle?" (default: all active)
2. **Loop**: "Auto-continue with /loop?" (suggest interval based on experiment speed)
3. **Priority overrides**: "Any branches to prioritize or skip?"

These preferences are logged in the handoff so subsequent cycles (or a new agent picking up from handoff) know the user's intent without re-asking.

## Sub-Agent Architecture (Advanced)

For larger research programs, specialized sub-agents handle different roles:

```
Orchestrator (the loop)
├── Branch Agents (run experiments, one per branch, parallel)
├── Red Team Agent (shuffled-label checks, every 5th cycle)
├── Archivist (maintains experiment history, dead ends)
└── Execution Agent (tests against live/production systems)
```

State files are the synchronization primitive. No message queue, no database, no framework.
